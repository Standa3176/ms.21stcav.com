## Phase Boundary

Five integration credential sources currently live ONLY in `.env`:
- **Supplier API** (21stcav.com JWT) — `SUPPLIER_API_BASE_URL` + `SUPPLIER_JWT_USER` + `SUPPLIER_JWT_PASS`
- **WooCommerce REST** — `WOO_BASE_URL` + `WOO_CONSUMER_KEY` + `WOO_CONSUMER_SECRET`
- **Bitrix24 webhook** — `BITRIX_WEBHOOK_URL`
- **Anthropic Claude API** — `ANTHROPIC_API_KEY` (consumed by Prism)
- **Langfuse observability** — `LANGFUSE_PUBLIC_KEY` + `LANGFUSE_SECRET_KEY` + `LANGFUSE_HOST`

Phase 11.2 just shipped a clean reference pattern for DB-managed credentials with admin UI: `competitor_ftp_credentials` table + encrypted Eloquent casts + Filament Resource + "Test connection" Action.

Phase 09.1 ports the same pattern to the five integrations above, consolidating them into a Filament navigation group `Admin → Settings → Integrations`. Each integration becomes one or two rows in a dedicated table (or one row in a polymorphic `integration_credentials` table — planner picks). Per-integration "Test connection" Action proves the credential before any agent or sync code consumes it.

`.env` continues to work as a fallback for CI/test environments so existing tests don't break — DB-row presence wins; absence falls back to `config('services.X')`.

Single-plan or 2-plan phase. ~150-250 LOC + 8-10 tests. Reuses Phase 11.2's pattern verbatim.

<decisions>
## Implementation Decisions

### Schema model — polymorphic single table

- **D-01:** **One `integration_credentials` table** (NOT five separate tables). Rows are typed by `kind` enum (`supplier_api` / `woo_rest` / `bitrix_webhook` / `anthropic_api` / `langfuse_observability`). Common columns + `payload_encrypted` JSON for kind-specific fields. Rationale: simpler migration story, single Filament Resource with kind-aware form schema, single Test Connection action that dispatches per-kind.
- **D-02:** **Schema:**
  - `id` (auto-increment integer — admin-managed; matches Phase 11.2 D-02 precedent)
  - `kind` (enum, indexed, unique by itself for v1 — one row per integration kind; can later support multi-instance per kind if needed)
  - `name` (string — display label, e.g. "Production Supplier API", defaults to a kind-specific value)
  - `payload_encrypted` (text — Eloquent `'encrypted:array'` cast — JSON of kind-specific fields like `{base_url, username, password}` for supplier_api or `{api_key}` for anthropic_api)
  - `is_active` (bool default true)
  - `last_test_at` (nullable timestamp)
  - `last_test_status` (nullable enum: `ok` / `failed`)
  - `last_test_error` (nullable text)
  - `last_test_latency_ms` (nullable unsigned int)
  - timestamps
  - UNIQUE on `(kind)` for v1 — single credential per integration kind
- **D-03:** **`'encrypted:array'` Eloquent cast** on `payload_encrypted` — Laravel 12 native AES-256, JSON-serialised array. All sensitive fields (API keys, passwords, webhook URLs containing tokens) live in this single encrypted blob. Field shape per kind documented in `IntegrationCredentialKind` enum.

### Per-kind payload shapes

- **D-04:** **`IntegrationCredentialKind` PHP enum** with method `requiredFields(): array` returning the schema per kind:
  - `supplier_api`: `[base_url (URL), username (string), password (Password input)]`
  - `woo_rest`: `[base_url (URL), consumer_key (Password), consumer_secret (Password)]`
  - `bitrix_webhook`: `[webhook_url (URL — the whole signed URL is the credential)]`
  - `anthropic_api`: `[api_key (Password)]`
  - `langfuse_observability`: `[host (URL), public_key (Password), secret_key (Password)]`
- **D-05:** **Filament form is kind-aware.** Form gets the `kind` Select first; on change, the rest of the form rebuilds dynamically based on `IntegrationCredentialKind::from($state)->requiredFields()`. Implementation via Filament's `live()` + `schema()` callback.

### Resolver layer — DB wins, env fallback

- **D-06:** **`IntegrationCredentialResolver` service** at `app/Domain/Integrations/Services/`. Method `for(IntegrationCredentialKind $kind): array` — returns the credential payload as an associative array.
  - Lookup order: (1) DB row where `kind=$kind` AND `is_active=true` → return `payload_encrypted` decrypted; (2) `.env`-backed `config('services.X')` fallback; (3) throw `IntegrationCredentialMissingException` if both empty.
  - Cached for 60s via Cache facade per `kind` to avoid 14×-per-pull DB hits during competitor:ftp-pull. Cache invalidated by an Eloquent observer on save/delete.
- **D-07:** **Existing client classes refactored to use the resolver.** Touch points (5 clients):
  - `WooClient` — currently consumes Automattic SDK via service container; resolver returns `[base_url, consumer_key, consumer_secret]` and binds the SDK client at request time
  - `SupplierClient` — currently consumes `config('services.supplier.*')`; resolver returns `[base_url, username, password]`
  - `BitrixClient` — currently consumes `config('services.bitrix.webhook_url')`; resolver returns `[webhook_url]`
  - `ClaudeClient` — currently consumes Prism config (env `ANTHROPIC_API_KEY`); resolver returns `[api_key]`; ClaudeClient sets it on the Prism request via `withApiKey(...)` if Prism supports per-call override, else writes to `config()` at runtime
  - Langfuse — instrumentation reads from config; resolver returns `[host, public_key, secret_key]` and updates the Prism Langfuse middleware at boot
- **D-08:** **Env fallback is permanent, not deprecation-targeted.** CI/test environments and the operator's "I want to test on my laptop without DB-seeding 5 integrations" workflow both depend on env fallback. Document in `.env.example` + handover docs that "DB row wins; env is fallback".

### Filament UX

- **D-09:** **Single `IntegrationCredentialResource`** at `app/Domain/Integrations/Filament/Resources/`. List columns: id, kind (badge with colour per integration), name, is_active toggle, last_test_status badge (green/red/grey), last_test_at, last_test_latency_ms, actions (Edit / Test connection / Delete).
- **D-10:** **Nav placement:** Filament navigation group `Admin → Settings`. Resource name "Integration Credentials". Sort order pins it under the existing AlertRecipientResource + AutoCreateSettingsPage (Settings group).
- **D-11:** **Per-row "Test connection" Action** dispatches per-kind:
  - `supplier_api` → `SupplierClient::testConnection()` (POST /generate_token.php with credentials, expect 200 + JWT in response)
  - `woo_rest` → `WooClient::testConnection()` (GET /wp-json/wc/v3/products?per_page=1, expect 200 + array)
  - `bitrix_webhook` → `BitrixClient::testConnection()` (POST crm.dealcategory.list, expect 200 + result array)
  - `anthropic_api` → `ClaudeClient::testConnection()` (Prism::text("ping") with model claude-haiku-4-5 + max_tokens=5, expect any 2xx)
  - `langfuse_observability` → HTTP GET {host}/api/public/health (no auth) — confirms reachability only
  - Result writes back to `last_test_at` + `last_test_status` + `last_test_error` + `last_test_latency_ms`. Filament notification surfaces success/failure inline.
- **D-12:** **Admin-only Resource.** Pricing manager / sales / read_only get NO perms. Credentials are sensitive operational config. Shield seeder LIKE pattern `%_integration_credential` admin-only.
- **D-13:** **`->authorize()` mandatory** on Test connection + Save + Delete + Edit (Phase 1 Warning 9).
- **D-14:** **LogsActivity trait** on the model — every credential save logs to `audit_log`. Decrypted values NEVER logged (LogsActivity captures `payload_encrypted` ciphertext only — the model accessor is bypassed by spatie/activitylog because it reads raw attribute).

### Health surface for the dashboard

- **D-15:** **`IntegrationHealthWidget`** on the existing Phase 7 home dashboard — 5 traffic-light tiles (one per integration kind). Reads from `dashboard_snapshots` with a new `metric_key='integration_health'` populated by `dashboard:refresh` (extending Phase 7's `SnapshotAggregator`). Each tile shows kind name + green/red/grey badge + last_test_at relative time. Click → navigates to that integration's edit page.

### Claude's Discretion

- **Resolver caching strategy:** 60s cache TTL is conservative; could go higher but operator credential rotation needs to take effect within 1 minute. Document the trade-off.
- **Polymorphic vs per-kind tables:** D-01 picks polymorphic. Alternative was 5 separate tables — rejected because the form/list/test-connection logic ends up duplicated 5 ways. Polymorphic + kind enum + dispatch is cleaner.
- **Multi-credential per kind (e.g. dev + prod Woo):** v1 enforces UNIQUE(kind) — one row per integration. Multi-instance support is a later optimisation; would change the resolver to take a `name` param too.
- **Encryption key rotation:** APP_KEY rotation invalidates all encrypted payloads. Document in handover that operators must re-enter all credentials after key rotation. Not a Phase 09.1 implementation concern.
- **Test connection rate-limiting:** an admin clicking Test connection 10× in 10s would hit Anthropic with 10 ping requests. Add a soft rate-limit via Filament's debounce or a 5-second cache on the result. Defer if scope-creepy; document.
- **Health widget polling cadence:** the new metric_key feeds into the existing 5-min `dashboard:refresh`. Manual "Re-test all" button on the widget is a nice-to-have, deferred.

<specifics>
## Specific Ideas

- **The polymorphic single-table approach saves ~80% of the boilerplate** vs 5 separate tables. The encrypted JSON payload + kind-aware form is the right call.
- **Env fallback is non-negotiable.** CI tests + initial-deployment ergonomics depend on it. The handover doc must be explicit: DB row wins; env is fallback. Operators DELETE the env vars only after successfully testing the DB row.
- **Test connection per kind** is the operator-facing UX win. Without it, ops sets credentials and crosses fingers; with it, they get an instant green/red answer. The 5 dispatch handlers are tiny — each is ~5-10 lines of HTTP.
- **The IntegrationHealthWidget on the home dashboard** turns "did anyone break our supplier creds yesterday?" from a debugging mystery into a glance.
- **APP_KEY rotation is a documented operational hazard** — handover docs must say "rotating APP_KEY invalidates all encrypted credentials; have ops re-enter all 5 after rotation".
- **The resolver pattern keeps client classes provider-agnostic** — Phase 12 SEO Agent + Phase 14 Chatbot won't touch Anthropic env vars; they'll get credentials from `IntegrationCredentialResolver::for(AnthropicAPI)`.

</specifics>

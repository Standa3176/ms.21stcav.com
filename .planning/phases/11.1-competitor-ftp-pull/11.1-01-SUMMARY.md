## D-10 RESOLUTION — csv_parse_errors.issue_type ENUM

The Phase 5 ENUM creates an SQLite-level CHECK constraint on testing DBs. The Phase 11.1 migration handles BOTH paths:

**MySQL:** `ALTER TABLE csv_parse_errors MODIFY COLUMN issue_type ENUM(...)` — extends the enum allow-list while preserving the DB-level guarantee.

**SQLite:** Drops the `csv_parse_errors_type_resolved_idx`, drops the column, re-adds with the extended enum list (which Laravel emits as a fresh CHECK constraint), then restores the index. RefreshDatabase test DBs are empty when this runs — no row migration needed.

Local execution against the testing DB was deferred (MySQL not running on the dev host: `127.0.0.1:3306` connection refused — Phase 6/7/10/11 deferral precedent). All Pest tests run against in-memory SQLite via `DB_CONNECTION=sqlite DB_DATABASE=:memory:` env override and pass against the SQLite ENUM-rebuild path.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Installed `league/flysystem-ftp` + `league/flysystem-sftp-v3` packages**
- **Found during:** Task 2 unit-test run (`Class "League\Flysystem\Ftp\FtpAdapter" not found`)
- **Issue:** Plan D-01 + Task 2 read_first claimed both packages were transitively pulled by Laravel 12. composer.lock listed them only as `suggest` — they were NOT installed.
- **Fix:** `composer require league/flysystem-ftp:^3.25 league/flysystem-sftp-v3:^3.25 --ignore-platform-req=ext-ftp --ignore-platform-req=ext-pcntl --ignore-platform-req=ext-posix` (dev box ext-ftp/pcntl/posix are missing; production VPS has them enabled — required at install-time).
- **Files modified:** composer.json, composer.lock
- **Commit:** b05a518

**2. [Rule 3 - Blocking] Removed `final` keyword from `FtpSourceConnector` class**
- **Found during:** Task 2 feature-test run (`Class ... cannot extend final class App\Domain\Competitor\Ftp\Services\FtpSourceConnector`)
- **Issue:** The plan-spec told tests to bind a stub via `$this->app->instance(FtpSourceConnector::class, $stub)` extending the connector. With `final` the test stub cannot extend the production class. Mockery would work but anonymous-class extension is the cleaner pattern and matches the plan's literal `<behavior>` example.
- **Fix:** Drop `final` from the class declaration; document the rationale in the class docblock so future maintainers know it's intentional (for testability, not extensibility).
- **Files modified:** `app/Domain/Competitor/Ftp/Services/FtpSourceConnector.php`
- **Commit:** b05a518

**3. [Rule 3 - Blocking] Made `FTP_BINARY` constant defensive**
- **Found during:** Task 2 unit-test run (`Undefined constant "App\Domain\Competitor\Ftp\Services\FTP_BINARY"`)
- **Issue:** `FTP_BINARY` lives in PHP's ext-ftp extension. Dev boxes without ext-ftp enabled (Herd Windows) hit a fatal error just LOADING the FtpSourceConnector class — even when only running SFTP-path code paths (or unit tests that introspect the SFTP adapter).
- **Fix:** Replaced bare `FTP_BINARY` with `defined('FTP_BINARY') ? FTP_BINARY : 2` (the literal value of the constant per ext-ftp source). Production VPS has ext-ftp enabled (`league/flysystem-ftp` requires it at composer install-time without `--ignore-platform-req`).
- **Files modified:** `app/Domain/Competitor/Ftp/Services/FtpSourceConnector.php`
- **Commit:** b05a518

**4. [Rule 3 - Blocking] SQLite drop+recreate fallback in ENUM-extension migration**
- **Found during:** Task 2 failure tests (`SQLSTATE[23000]: Integrity constraint violation: 19 CHECK constraint failed: issue_type`)
- **Issue:** Plan A2 said the SQLite path should be a "silent no-op" — but Laravel's `$t->enum(...)` on SQLite emits a CHECK constraint that does NOT include 'ftp_pull_failed'. Tests against in-memory SQLite (the only DB option since MySQL was offline on the dev host) hit a constraint violation when inserting `ftp_pull_failed` rows.
- **Fix:** Extended the migration with a `$driver === 'sqlite'` branch that drops the index → drops the column → re-adds with the extended enum allow-list → restores the index. RefreshDatabase test DBs are empty when this runs, so no row data preservation needed.
- **Files modified:** `database/migrations/2026_05_02_010100_add_ftp_pull_failed_to_csv_parse_errors_issue_type.php`
- **Commit:** b05a518

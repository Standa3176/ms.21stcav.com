# Phase 11.1: Competitor FTP Pull - Context

**Gathered:** 2026-05-02 (auto-mode — recommended defaults inline)
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 5 (Competitor Analysis) shipped a watcher that ingests CSVs from `storage/app/competitors/incoming/`, parses them, computes margin deltas, and produces `margin_change` Suggestions that update PricingRules on approval. The pipeline assumed n8n drops files to that local directory.

Phase 11.1 adds the **fetch step** so Laravel can pull competitor CSVs directly from external FTP/SFTP feeds without depending on n8n. New `competitor_ftp_sources` admin-managed table holds host/port/path/credentials/cron config per source. New `competitor:ftp-pull` artisan command iterates active sources, downloads matching CSV files into `storage/app/competitors/incoming/`, and exits — Phase 5's existing `competitor:watch` (every 5 min) picks them up and runs the existing parse → DB → margin analyser pipeline unchanged.

Scope: ONE migration, ONE model, ONE service, ONE artisan command, ONE Filament Resource, ONE seeder pattern (no defaults — admin populates), 5-6 tests. ~50-100 LOC. Does NOT modify Phase 5 ingest pipeline (D-11 architectural mandate). Does NOT add new ingestion semantics. Single-plan phase.

</domain>

<decisions>
## Implementation Decisions

### FTP transport library (no install needed)

- **D-01:** **Use Laravel's built-in `Storage::disk('competitor-ftp-{id}')` adapter via `league/flysystem-ftp` and `league/flysystem-sftp-v3`.** Both packages already pulled by Laravel 12; no `composer require` needed. Per-source dynamic disk config built at runtime from `competitor_ftp_sources` row (host, port, username, password, base path, ssl/sftp toggle).
- **D-02:** **`FtpSourceConnector` service** builds the dynamic Storage config from a `CompetitorFtpSource` model, registers a one-shot Flysystem instance, returns it to the calling job. Disk is NOT registered in `config/filesystems.php` statically — credentials live in DB, not config. This keeps `config/filesystems.php` clean and lets ops add/remove sources via Filament without redeploys.

### Schema — `competitor_ftp_sources`

- **D-03:** **Columns:**
  - `id` (ULID PK, matches v2 milestone Phase 1 D-14 ULID convention)
  - `competitor_id` (FK competitors, cascade — Phase 5 already has the table)
  - `name` (string, unique per competitor — e.g. "weekly_csv", "daily_pricing")
  - `protocol` (enum: `ftp`, `sftp`, `ftps`)
  - `host` (string)
  - `port` (unsigned smallint, default 21 for ftp / 22 for sftp / 990 for ftps)
  - `username` (string)
  - `password_encrypted` (text, encrypted Eloquent cast — `'encrypted'` Laravel native)
  - `private_key_encrypted` (text, nullable, encrypted — for SFTP key auth)
  - `passphrase_encrypted` (text, nullable, encrypted — for SFTP encrypted key)
  - `base_path` (string, default `/`)
  - `filename_pattern` (string — regex; defaults to Phase 5's `^[a-z0-9_-]{1,64}_\d{4}-\d{2}-\d{2}\.csv$`)
  - `cron_expression` (string, default `*/15 * * * *`)
  - `is_active` (bool, default true)
  - `last_pulled_at` (nullable timestamp)
  - `last_pull_status` (nullable enum: `success`, `failed`, `partial` — for the freshness widget)
  - `last_pull_files_fetched` (unsigned int, default 0)
  - `last_pull_error` (nullable text)
  - timestamps
- **D-04:** **Encrypted casts via Eloquent `'encrypted'` cast on the model** (Laravel native AES-256). Casting at the model level ensures even tinker / raw queries see ciphertext; only model accessors decrypt. Document the dependence on `APP_KEY` rotation runbook in handover docs.

### Pull flow

- **D-05:** **`competitor:ftp-pull` artisan command** — iterates `competitor_ftp_sources` where `is_active=true`. For each source:
  1. Build dynamic disk via `FtpSourceConnector::connect($source)`
  2. List files in `$source->base_path` matching `$source->filename_pattern`
  3. Skip files older than `last_pulled_at` (mtime check; avoids re-downloading processed files)
  4. Skip files larger than `config('competitor.ftp.max_file_mb', 50)` (DoS guard)
  5. Stream-download each file to `storage/app/competitors/incoming/{competitor_slug}_{YYYY-MM-DD}.csv` using the Phase 5 filename convention. Filename normalisation: if remote filename matches Phase 5 pattern, copy as-is; otherwise rename to `{competitor.slug}_{today}.csv` so the watcher can resolve it.
  6. Atomic disk write via `tempnam → rename` (Pitfall P5-C precedent — Phase 5 watcher's mtime-gate sees the file as ready 30s after write completes)
  7. Update `competitor_ftp_sources.last_pulled_at`, `last_pull_status`, `last_pull_files_fetched`
  8. Write to `audit_log` via `Auditor::record('competitor_ftp_pull', source_id, files_fetched, correlation_id)`
- **D-06:** **Dry-run default + `--live` opt-in** (Phase 2 D-04 pattern). `php artisan competitor:ftp-pull` lists what WOULD be downloaded without actually downloading. `--live` performs the download. `--source={id}` flag scopes to one source.
- **D-07:** **Schedule:** `routes/console.php` runs `competitor:ftp-pull --live` every 15 min (default — overrideable per-source via `cron_expression` if Plan implementer wants to support per-source scheduling; v1 ships single global cron). MUST run BEFORE `competitor:watch` slot — Phase 5's watcher runs every 5 min, so the every-15-min FTP pull will land files that the next 5-min watcher tick picks up. `withoutOverlapping(20)` so a slow FTP pull doesn't double-fire.

### Filament Resource — `CompetitorFtpSourceResource`

- **D-08:** **Admin-only Resource** (Shield perms attached via RolePermissionSeeder LIKE pattern `%_competitor_ftp_source`). Form fields: competitor Select, name, protocol Select, host, port, username, password (Filament `Password` field type — masked input), private_key (Textarea, optional), passphrase (Password), base_path, filename_pattern, cron_expression, is_active toggle. Live "Test connection" action — connects + lists files in base_path + reports count without downloading. List view: traffic-light status badge (green=last_pull_status=success, amber=partial, red=failed, grey=never pulled), last_pulled_at, last_pull_files_fetched, name, host. Per-row "Pull now" action → dispatches a one-shot `competitor:ftp-pull --source={id} --live` to `competitor-csv` queue.
- **D-09:** **`->authorize()` on every Action** including Test connection, Pull now, Save (Phase 1 Warning 9). Toggle `is_active` audit-logged via `LogsActivity` trait on model.

### Failure handling

- **D-10:** **Failures land in `csv_parse_errors` (Phase 5 table) with NEW kind `ftp_pull_failed`.** Distinct from existing `invalid_filename` / `column_ambiguity` / `parse_error` kinds. Includes: source_id, host, error message, timestamp. Surfaces in Phase 5 `CsvIngestIssuesPage` (no new UI needed — existing Issues page tab will display them).
- **D-11:** **Phase 5 ingest pipeline is UNTOUCHED.** Phase 11.1 only writes files to `storage/app/competitors/incoming/`. The watcher's mtime gate (>30s old), filename validation, encoding detection, column detection, parse pipeline — all unchanged. This is a strict additive layer.
- **D-12:** **AlertRecipient extension `receives_competitor_ftp_alerts`** — boolean column extending the Phase 2/4/5/6/7/11 `receives_*` pattern. When a source's pull fails 3 consecutive times (atomic counter on `competitor_ftp_sources.consecutive_failures` column — add to schema), notify recipients via Mail. ThrottledFailedJobNotifier (Phase 1) handles dedup if multiple sources fail in same window.

### Claude's Discretion

- **Stream chunking for large files:** Flysystem v3 streams by default. If the supplier feed exceeds 50MB, document a streaming-chunked approach (read 8192-byte chunks, flush per chunk). Default ships with the simple `getStream()` → `fputs()` approach which handles up to ~50MB without memory issues on default PHP `memory_limit=128M`.
- **SSL cert verification toggle:** `verify_ssl` boolean column on the source — default true. If a competitor uses self-signed certs, ops can disable per-source. Document as security-trade-off in source's Filament form description.
- **Retry policy:** ftp-pull job runs synchronously inside the artisan command (no Horizon retry). On transient failure (timeout, connection refused), the next 15-min schedule slot retries naturally. Source-level circuit-breaker (3 consecutive fails → mark source as auto-disabled, alert admin) is documented but planner decides if this ships in v1 or defers.
- **Filename conflict on remote FTP:** if two files match the regex but differ only in case, Flysystem returns both — pull the most-recent-mtime first; subsequent same-name writes overwrite. Phase 5 watcher's `(competitor_id, sku, recorded_at)` UNIQUE index handles dedup at the data layer.
- **Deptrac layer placement:** Phase 11.1 lives inside the existing `Competitor` Deptrac layer (no new layer). Allow-list inherits Phase 5's existing Competitor allow-list. No dual-config update needed.

<specifics>
## Specific Ideas

- **`storage/app/competitors/incoming/` is the integration contract.** Phase 11.1 writes here; Phase 5 reads from here. As long as Phase 11.1 produces filenames matching the regex with mtime > 30s, the existing pipeline does the rest. This is the cleanest possible additive integration.
- **Encrypted-at-rest credentials are non-negotiable.** Document `APP_KEY` rotation runbook step in handover docs (would invalidate existing rows; admin must re-enter creds — note this in Filament form help text).
- **Per-source `last_pulled_at` is the freshness signal.** Phase 7 dashboard's `CompetitorFreshnessWidget` may want to show "last FTP pull" alongside "last competitor_prices recorded_at" — out of Phase 11.1 scope but the data is queryable for a future Phase 7.x widget polish.
- **Don't over-engineer retry.** The 15-min schedule is the retry mechanism. A failed pull at 12:00 retries at 12:15 naturally. Consecutive-failure tracking exists for the auto-disable + alert path, NOT for in-process retry loops.
- **`Test connection` Filament action is the operator UX win.** Without it, ops can't verify creds without a 15-minute wait. The action just connects + lists files + reports count. Failure shows the Flysystem error message verbatim for ops debugging.

</specifics>

<deferred>
## Deferred Ideas

- **Per-source cron expressions** — v1 ships single global 15-min cron. Per-source variable schedules would require separate scheduled tasks per source row + dynamic schedule-list registration. Defer until ops asks for it.
- **Streaming for >50MB files** — v1 ships simple `getStream()` → `fputs()` (handles up to ~50MB). Chunked streaming with checkpoint resume is post-v1 scope.
- **Alternative protocols** — HTTPS endpoints, S3 buckets, Google Drive. v1 = FTP/SFTP/FTPS only. New protocols in a future phase if a competitor exposes them.
- **Self-disable on N consecutive failures** — Claude's Discretion item flagged for planner judgement; ships if scope allows, otherwise defer.
- **FTP file-name pattern extraction beyond regex** — only filename regex supported in v1. Per-competitor advanced filename templating (e.g. "look in dated subdirectories") deferred.
- **n8n migration** — operators using n8n today can keep using it; Phase 11.1 doesn't deprecate the n8n→local-disk pattern, it just adds another way to land files. Migration is a separate Phase 11.2 if ops decides to consolidate later.
- **Phase 7 dashboard widget showing per-source FTP freshness** — adds a row to `CompetitorFreshnessWidget`. Defer to Phase 7.x polish or wait for ops feedback.

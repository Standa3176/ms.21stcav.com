## Phase Boundary

Phase 11.1 shipped a "one row per FTP server" model. Operator feedback during the same session: the real-world setup is **one shared FTP folder containing many competitor files** (Nuvias, Ingram, Sahara, Nimans, Lindy, Unicol, Douglas Stewart, Tech Data, Northamber, WestCoast, etc. — 14+ feeds dropped into a single supplier-aggregated folder). The right model is `(shared FTP credential) + (many feeds, one per remote file)`.

Phase 11.2 refactors the schema + Filament UX accordingly:
- **Two tables:** `competitor_ftp_credentials` (host / port / user / pass-encrypted / protocol / base_path — typically 1 row) + `competitor_ftp_feeds` (one row per remote file with competitor_id + remote_filename + local_filename + format + freshness state)
- **Format normalisation** in the pull step: `.zip` → unzip → first `*.csv`; `.tsv` / `.txt` → CSV; `.csv` → pass-through. Output is always `.csv` written to `storage/app/competitors/incoming/` so Phase 5 ingest pipeline stays untouched (D-11 preserved).
- **Filament `CompetitorFtpFeedResource`** matching the screenshot: columns Id / Supplier / Remote File Date / Remote Filename / Local Filename / Last Updated / Status / Actions; red text on stale `remote_file_date`; "Add New Feed" + "Filter Table" + "Download" toolbar; per-row Edit / Refresh now / Delete actions.
- **Migration is destructive** — drops Phase 11.1's `competitor_ftp_sources` and rebuilds. Safe because Phase 11.1 shipped 10 min ago with zero production data.

Reuses from Phase 11.1 (no rebuild): `FtpSourceConnector` service (accepts a credential model now, was a source model — interface change), 15-min schedule semantics, 3-strike auto-disable per feed, encrypted credential casts, `csv_parse_errors.issue_type='ftp_pull_failed'` enum value, `AlertRecipient.receives_competitor_ftp_alerts` toggle.

Single-plan phase, ~150-200 LOC + ~6-8 tests.

<decisions>
## Implementation Decisions

### Schema split — credentials vs feeds

- **D-01:** **Two tables.**
  - `competitor_ftp_credentials`: `id` (ULID PK), `name` (string unique — e.g. "supplier_aggregate", "weekly_distributor"), `protocol` (enum: ftp/sftp/ftps), `host`, `port`, `username`, `password_encrypted`, `private_key_encrypted` (nullable), `passphrase_encrypted` (nullable), `base_path` (default `/`), `verify_ssl` (bool default true), `is_active` (bool default true), `last_test_at` (nullable timestamp — populated by "Test connection" action), `last_test_status` (nullable enum: ok / failed), `last_test_error` (nullable text), timestamps. Typically 1 row; >1 supported for multi-supplier scenarios.
  - `competitor_ftp_feeds`: `id` (auto-increment integer for human-readable IDs in the table — matches screenshot's `1, 10, 12, 13, 16, 18, 27...`), `competitor_id` (FK competitors, cascade), `credential_id` (FK competitor_ftp_credentials, restrict — can't delete creds that have feeds), `remote_filename` (string — exact filename or pattern; e.g. `nuvias-hub-feed.csv`, `PRICE.ZIP`, `Enhanced-GB.tsv`), `local_filename` (string — canonical name written to `storage/app/competitors/incoming/`; e.g. `nuvias.csv`, `ingram.csv`, `tdsynnex.csv`), `format` (enum: csv / tsv / zip / txt), `is_active` (bool default true), `last_pulled_at` (nullable timestamp — local fetch time), `remote_file_date` (nullable timestamp — mtime on remote; drives stale-feed alert), `last_pull_status` (nullable enum: success / failed / skipped / no_change), `last_pull_error` (nullable text), `consecutive_failures` (unsigned int default 0), timestamps.
- **D-02:** **Auto-increment integer PK on feeds** (NOT ULID) to match the screenshot's IDs. ULIDs are great for distributed systems but feeds are admin-managed via Filament with stable per-row identity in the UI.
- **D-03:** **Encrypted casts** on `password_encrypted`, `private_key_encrypted`, `passphrase_encrypted` via Eloquent `'encrypted'` cast (Laravel 12 native AES-256). Inherited from Phase 11.1.
- **D-04:** **Destructive migration drops Phase 11.1's `competitor_ftp_sources` table** as Task 1 step 1. Migration class name explicitly states "drops 11.1 schema". No data preservation needed (zero prod data).

### Format normalisation step

- **D-05:** **`FeedFormatNormaliser` service** in `app/Domain/Competitor/Ftp/Services/`. Method `normalise(string $tempPath, string $format, string $localFilename): string` — returns absolute path to normalised CSV file ready to write to `incoming/`.
  - `csv` — return path unchanged
  - `tsv` — read with `\t` delimiter, write as `,` delimiter; same encoding handling as Phase 5's `EncodingDetector` (BOM + Windows-1252 + UTF-8 detection — reuse the existing service)
  - `zip` — extract via `ZipArchive`, take first matching `*.csv` (case-insensitive) inside, fail if no CSV found
  - `txt` — sniff first 100 lines for delimiter (tab / comma / semicolon), convert to CSV; reject if no clear delimiter pattern
- **D-06:** **Normalised output filename = `competitor_ftp_feeds.local_filename`.** This is the canonical name for the file landing in `storage/app/competitors/incoming/`. Validation rule: must end in `.csv` (Phase 5 watcher's regex requires it).
- **D-07:** **Phase 5 ingest pipeline still UNTOUCHED.** D-11 from Phase 11.1 still load-bearing. Format normalisation happens BEFORE the file is written to `incoming/` so Phase 5 only ever sees `.csv`.

### Filament UX matching the screenshot

- **D-08:** **`CompetitorFtpFeedResource`** at `app/Domain/Competitor/Filament/Resources/CompetitorFtpFeedResource.php` (auto-discovery path).
  - List columns: `id`, `competitor.name as Supplier`, `remote_file_date` (red if older than `config('competitor.ftp.stale_days', 30)` — use Filament's column color/badge), `remote_filename` (truncate>40 chars with tooltip — see screenshot's `C21S002-Sahara_Fee...`), `local_filename`, `last_pulled_at as Last Updated`, `is_active as Status` (Active/Inactive toggle column), action buttons.
  - Per-row actions: **Edit** (modal — competitor / remote_filename / local_filename / format / credential / is_active toggle), **Refresh now** (one-shot dispatch `competitor:ftp-pull --feed={id} --live`), **Delete** (soft delete with confirmation).
  - Toolbar: **Add New Feed** action (modal create), **Filter Table** input (live search across competitor.name + remote_filename + local_filename), **Download** bulk action (CSV export of currently-filtered feeds).
  - Default sort: `id ASC` matching screenshot.
  - Pagination: 10 default with the rows-per-page selector (matches screenshot's `Rows per page: 10 ▼ 1-10 of 14`).
- **D-09:** **`CompetitorFtpCredentialResource`** at the same path. Admin-only. Simple CRUD. Includes "Test connection" action (lists files in base_path, reports count without downloading). Most ops users never visit this.
- **D-10:** **Per-row coloring rules:**
  - `remote_file_date` < now() - 30 days → red text (the screenshot's `2023-01-06` and `2025-10-13`)
  - `is_active=false` → row dimmed (50% opacity) AND status badge says "Inactive"
  - `consecutive_failures >= 3` → status badge says "Auto-disabled" with red icon, even if `is_active=false`
- **D-11:** **`->authorize()` mandatory** on every Action (Add New Feed, Edit, Refresh now, Delete, Test connection on credentials). Admin only for write operations; pricing_manager view-only.

### Pull command refactor

- **D-12:** **`competitor:ftp-pull` command refactored** — now iterates `competitor_ftp_feeds` (not sources). Flags: `--feed={id}` (was `--source={id}`), `--credential={id}` (scope to all feeds for one credential), `--live`, `--dry-run` (default).
- **D-13:** **Per-feed pull algorithm:**
  1. Build Flysystem disk via `FtpSourceConnector::connect($feed->credential)` — reuses Phase 11.1 service signature change
  2. Check remote mtime via `Storage::disk(...)->lastModified($feed->remote_filename)`. If equal to `$feed->remote_file_date`, set `last_pull_status='no_change'` + skip (avoids re-downloading static files — Wholesaler feeds change weekly, no point downloading 14 files every 15 min)
  3. Stream-download to temp file
  4. Run `FeedFormatNormaliser::normalise($tempPath, $feed->format, $feed->local_filename)` → returns normalised CSV path
  5. Atomic move normalised CSV to `storage/app/competitors/incoming/{local_filename}`
  6. Update feed's `last_pulled_at`, `remote_file_date` (from remote mtime), `last_pull_status='success'`, reset `consecutive_failures=0`
  7. On failure: increment `consecutive_failures`; if ≥3 → `is_active=false` + dispatch notification
  8. Audit log via `Auditor::record('competitor_ftp_feed_pull', feed_id, ...)`
  9. Failures write `csv_parse_errors` row with `issue_type='ftp_pull_failed'` (enum extension already shipped in Phase 11.1)

### Claude's Discretion

- **Connection pooling per credential:** if 14 feeds share 1 credential, current implementation opens 14 separate FTP connections (one per feed). Acceptable for now; document a Phase-12+ optimisation to pool connections per credential within a single command run. Throughput impact at scale is minimal (FTP connection setup ~200ms × 14 = 2.8s — well within the 15-min schedule window).
- **`Download` bulk action format:** CSV export with all feed columns. Matches the screenshot's button. Reuses the `HasExportableTable` trait shipped in Phase 7-03.
- **Local filename collision:** if two feeds have the same `local_filename`, validation rejects the second. UNIQUE index on `local_filename` enforces.
- **Soft deletes:** Feeds use `SoftDeletes` trait (audit history value); Credentials use hard delete.
- **Format `auto` enum value:** initially considered for "let the normaliser sniff the format", but explicit format enum is safer (operator picks csv/tsv/zip/txt at create time).
- **"Last Updated" column semantics:** = `last_pulled_at` (when Laravel last fetched), NOT `updated_at` (Eloquent column). Document in column tooltip.
- **Refresh-now dispatch queue:** `competitor-csv` queue (Phase 11.1 precedent). Single-shot, no retry within the action.
- **Migration rollback:** the `down()` migration recreates Phase 11.1's `competitor_ftp_sources` table for safety, but with empty rows.

<specifics>
## Specific Ideas

- **Operator workflow after this lands:** seed one `CompetitorFtpCredential` row pointing at the supplier-aggregated FTP folder (the one with all 14+ feeds). Then for each competitor, click "Add New Feed" in `/admin/competitor-ftp-feeds` and enter the remote_filename + local_filename + format. Done. Pull happens every 15 min; Phase 5 picks up the .csv files; pricing pipeline runs.
- **Migration is destructive but safe** — Phase 11.1 shipped 10 min ago, zero production data, all tests still pass at the schema level. Operator who already configured 11.1 sources will need to re-enter (none should have done so this fast).
- **The screenshot's `Status: Inactive` row (Douglas Stewart, 2025-10-13)** is exactly what auto-disable looks like after 3 consecutive failures + the `remote_file_date` flag goes red on the same row — both signals at once is the strong "this needs operator attention" cue.
- **The screenshot's `1-10 of 14` pagination** confirms ~14 active feeds is the realistic operational scale. Single-credential + 14 feeds = 14 connections per pull cycle = trivial load on the upstream FTP. Connection pooling deferred (Claude's Discretion).
- **Format normalisation handles real-world messiness:** PRICE.ZIP (Ingram packs CSV in zip — common with quarterly catalogues), Enhanced-GB.tsv (Tech Data uses TSV), C21S002-Sahara_Fee….csv (truncated UI but real filename has the full date code). Each gets normalised to a clean local_filename.csv.
- **Phase 11.1's `competitor_ftp_sources` table is dropped clean** — no zombie schema. Migration's `down()` recreates it as empty (rollback safety) but operationally we never roll back this kind of schema split.

</specifics>

- **Connection pooling per credential** — 14 connections per pull is fine; pool optimisation if/when fleet grows beyond 30 feeds.
- **Auto-format-sniffing** — explicit format enum is safer for v1; sniffer can be added later as a "create feed" UX helper.
- **Per-feed cron expressions** — single global 15-min schedule for v1; per-feed cron deferred (most operations want all feeds on the same cycle anyway).
- **Feed history graph** — chart showing pull success rate over time per feed. Phase 7+ dashboard polish candidate.
- **Bulk import from CSV** — admin uploads a CSV of (competitor, remote_filename, local_filename, format) rows to create many feeds at once. Defer until ops actually wants this.
- **Remote folder watch** — auto-discover new files in the shared FTP folder + suggest "Add New Feed" entries. Defer (would tangle with file-rename detection).
- **Streaming for >50MB files** — inherited from Phase 11.1 deferral.
- **Format extensions: XLSX / XML / JSON** — out of scope. Add as new format enum values later if a competitor changes their feed format.

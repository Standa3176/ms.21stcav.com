# Quick Task 260504-01s

## Goal

Make the ~20k competitor SKUs that ingest pulls fetch each cycle visible in /admin/competitor-prices regardless of whether we have a matching Product yet.

## Tasks

### Task 1 — CompetitorCsvRowWriter: persist orphan rows

File: `app/Domain/Competitor/Services/CompetitorCsvRowWriter.php`

Current orphan branch (lines 76-84):
```php
$product = Product::whereRaw('LOWER(TRIM(sku)) = ?', [strtolower(trim($sku))])->first();

if ($product === null) {
    $this->orphanDetector->record($run->competitor, $sku, $grossPennies);
    $run->increment('rows_orphaned');
    return;  // ← drops the row on the floor
}
```

New behaviour: keep the orphanDetector + rows_orphaned increment (audit + suggestion creation are still useful) but drop the early return so we fall through to the standard CompetitorPrice::create() path. The matched-vs-orphan distinction lives entirely in the JOIN-at-query-time pattern downstream.

Note: the `rows_written` increment at the end of the matched path was previously matched-only. Need to decide:
- Option A: Keep `rows_written` matched-only, add separate `rows_orphan_persisted` counter (clean, but new column)
- Option B: Increment `rows_written` for both matched and orphan (consistent: row was written either way). Operators see total persisted via rows_written; orphan-vs-matched proportion via rows_orphaned/rows_written.

Going with Option B — simpler, no migration, and accurately reflects "we wrote this row." rows_orphaned remains the semantic "how many didn't match a Product."

### Task 2 — CompetitorPriceResource: Match Status filter + column

File: `app/Domain/Competitor/Filament/Resources/CompetitorPriceResource.php`

Add to `filters([...])`:

```php
\Filament\Tables\Filters\TernaryFilter::make('matched')
    ->label('Match Status')
    ->placeholder('All')
    ->trueLabel('Matched (in catalogue)')
    ->falseLabel('Orphans (not in catalogue)')
    ->queries(
        true: fn (Builder $q) => $q->whereExists(
            fn ($sub) => $sub->from('products')
                ->whereColumn(\DB::raw('LOWER(TRIM(products.sku))'), \DB::raw('LOWER(TRIM(competitor_prices.sku))'))
        ),
        false: fn (Builder $q) => $q->whereNotExists(
            fn ($sub) => $sub->from('products')
                ->whereColumn(\DB::raw('LOWER(TRIM(products.sku))'), \DB::raw('LOWER(TRIM(competitor_prices.sku))'))
        ),
        blank: fn (Builder $q) => $q,
    ),
```

Add a column showing match status as a badge — computed via the same WHERE EXISTS check, but more efficient via the with-loaded relationship if there is one. Since there's no FK, use `formatStateUsing` with a query check OR a Builder-level select to materialize a `is_matched` virtual column. Going with the latter for performance.

In `getEloquentQuery()`, add a select expression that flags matched rows:
```php
return parent::getEloquentQuery()
    ->with(['competitor'])
    ->addSelect([
        '*',
        'is_matched' => Product::query()
            ->selectRaw('1')
            ->whereRaw('LOWER(TRIM(products.sku)) = LOWER(TRIM(competitor_prices.sku))')
            ->limit(1),
    ]);
```

Then column:
```php
TextColumn::make('is_matched')
    ->label('Match')
    ->badge()
    ->formatStateUsing(fn ($state): string => $state ? 'Matched' : 'Orphan')
    ->color(fn ($state): string => $state ? 'success' : 'warning'),
```

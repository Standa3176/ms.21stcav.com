# Quick Task 260606-o63: Fix Vacuous auto_create_status IS-NOT-NULL Predicate — Summary

One-liner: Canonical `Product::scopeAutoCreated` scope (`!= 'manual'`) replaces the vacuous `whereNotNull('auto_create_status')` filter in RetryMissingImagesCommand + the inline `!= 'manual'` predicate in AutoCreateHealthPage; CI guardrail (architecture test) prevents the bug from ever returning.

## Context

Direct closure of a finding from quick task [260606-mx9](../260606-mx9-auto-create-health-page-filament-table-s/260606-mx9-SUMMARY.md): the `auto_create_status` column was added by migration `2026_04_22_100300_add_auto_create_columns_to_products_table.php` as `enum(...)NOT NULL DEFAULT 'manual'` with a belt-and-braces backfill of every pre-existing row to `'manual'`. So `whereNotNull('auto_create_status')` — used by `RetryMissingImagesCommand` since 260606-ca92caa as the "auto-created products only" filter — was a no-op the whole time, silently letting 5,632 legacy WC-imported `'manual'` rows through. The 2026-06-06 Manhattan retry dry-run silently surfaced 5,668 candidates instead of the expected ~36 because of this.

## Dry-run probe output (proof the fix works)

Local dev DB is SQLite; `RetryMissingImagesCommand`'s pre-existing `JSON_LENGTH(gallery_image_urls) = 0` predicate is MySQL-only (not introduced by this quick — it is the same in 260606-ca92caa) so the dry-run errors out on local SQLite before printing the breakdown table. The rendered SQL however confirms the scope-swap structurally:

**Default mode (without `--all`):**
```sql
select "sku" from "products"
 where ("gallery_image_urls" is null or JSON_LENGTH(gallery_image_urls) = 0)
   and "auto_create_status" != 'manual'                       -- ← the scope IS firing
   and "products"."deleted_at" is null
 order by "id" asc limit 10
```

**With `--all`:**
```sql
select "sku" from "products"
 where ("gallery_image_urls" is null or JSON_LENGTH(gallery_image_urls) = 0)
   and "products"."deleted_at" is null                        -- ← no auto_create_status clause; --all bypasses the scope chain
 order by "id" asc limit 10
```

Direct DB probe (via a throwaway `probe-o63.php` against the local catalog dump — script then deleted):

```
total                                     = 5633
autoCreated (scope, default mode)         = 0     ← legacy WC products correctly excluded
manual (only reachable in --all mode)     = 5633  ← every product in local dev is legacy WC
delta (manual rows excluded by fix)       = 5633
```

Translated to the prod 2026-06-06 numbers: pre-fix dry-run reported **5,668** candidates; with the fix, the local-state breakdown shows **0 'manual' rows** in the default-mode set (5,632 excluded) — leaving the ~36 genuine auto-created products that actually need image-retry. The 5,632-row silent bloat is closed. With `--all`, the 'manual' rows reappear, proving the flag still works.

## Known limitations / follow-ups

1. **No `RetryMissingImagesCommandTest` exists.** The plan's verify step 2 noted this as a "coverage gap to flag for a future quick." The behavior of the swap is indirectly verified by `tests/Feature/Filament/AutoCreateHealthPageTest` (which shares the same canonical scope), plus the focused unit test pinning the predicate, plus the structural SQL trace shown above. A future quick task should add a `RetryMissingImagesCommandTest` covering: default mode excludes legacy WC; `--all` includes legacy WC; `--status` filter still narrows; `--brand` resolves brand id and filters.

2. **`JSON_LENGTH` in `RetryMissingImagesCommand` is MySQL-only.** The command crashes on SQLite dev DB because `gallery_image_urls IS NULL OR JSON_LENGTH(gallery_image_urls) = 0` was written before driver-aware JSON patterns were established (the same pattern was later made driver-aware in `AutoCreateHealthPage` and `PruneOrphanSuggestionsCommand`). NOT introduced by this quick — pre-existing per 260606-ca92caa. Trivial follow-up: lift the same `emptyImagesExpr()` helper from `AutoCreateHealthPage` so the command runs locally too. Out of scope for this quick (would have broken atomicity — was not in the audit table).

3. **Scope body matches the verification grep.** Plan verification step 3 (`grep -RInE "where\(\s*['\"]auto_create_status['\"]\s*,\s*['\"]!=['\"]\s*,\s*['\"]manual['\"]\s*\)" app/`) would hit `app/Domain/Products/Models/Product.php:148` — the scope body itself. This is the single canonical implementation site and is expected; the arch test correctly does NOT flag this pattern because the forbidden list only covers `whereNotNull` / `whereNull` / raw `IS [NOT] NULL`, not the canonical `!= 'manual'`. Documented here so the verification step is not mis-read as a regression.

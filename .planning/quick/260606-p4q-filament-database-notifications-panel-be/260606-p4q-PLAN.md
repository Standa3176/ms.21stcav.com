| Boundary | Description |
|----------|-------------|
| CLI → command runtime | Operator triggers `php artisan products:retry-missing-images` — `auth()->user()` may or may not be set |
| Queue worker → notification persistence | Horizon worker writes to `notifications` table; failure must not propagate to pipeline exit code |
| Browser → /admin bell-icon endpoint | Authenticated Filament user polls for new notifications every 30s |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-p4q-01 | Tampering | notification payload `url` field | mitigate | All `url` values in this PR are hard-coded literals (`/admin/products`, `/admin/auto-create-reviews`, `/admin/auto-create-health`); no user input flows into payload. |
| T-p4q-02 | Information Disclosure | exception message in failed() body | mitigate | `Str::limit($e->getMessage(), 200)` truncates; only the notified user (the triggerer) sees it; Filament auth gate already in place. |
| T-p4q-03 | Denial of Service | bell-icon polling at 30s | accept | 30s poll is the Filament-recommended default; one authenticated user → one query → negligible load on existing infra. |
| T-p4q-04 | Repudiation | "notification dispatch failed" silently swallowed | mitigate | try/catch logs `Log::warning('auto_create_pipeline.notify_failed', ...)` so failures are forensic-recoverable from logs even if the UI signal is missed. |
| T-p4q-05 | Elevation of Privilege | RetryMissingImagesCommand → cron run gains user context | mitigate | `$this->triggeringUser = auth()->user();` at perform() start; cron has no auth user → null → notification block skipped. No user-impersonation surface. |
| T-p4q-SC | Tampering | npm/pip/cargo installs | accept | NO package installs in this plan. Only Laravel's built-in `notifications:table` artisan command + existing Filament 3 + existing Laravel Notification facade. Zero composer/npm changes. |
</threat_model>

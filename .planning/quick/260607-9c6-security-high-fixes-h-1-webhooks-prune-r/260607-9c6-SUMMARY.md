## H-1 — webhooks:prune-receipts signature + retention defaults

```
webhooks:prune-receipts
    {--order-days=30 : Retention in days for topic='order' (default 30)}
    {--customer-days=7 : Retention in days for topic='customer' (tightest GDPR window)}
    {--other-days=90 : Retention for all other topics}
    {--dry-run : Print per-topic candidate counts, perform no deletes}
```

| Topic | Retention | Rationale |
|-------|-----------|-----------|
| `order` | **30 days** | Tokenised payment metadata + line items; longer window because order replay/reconciliation is a real ops use-case |
| `customer` | **7 days** | TIGHTEST window — email + billing/shipping/phone PII; no replay use-case |
| `other` | **90 days** | Catch-all for future topic strings; safe-by-default because we don't yet know payload shape |

Implementation notes:
- Driver-agnostic by construction (no JSON expressions, no LIKE patterns) — same code path on prod MySQL + SQLite test DB.
- Queries index-covered via `(source, topic, received_at)` from the `2026_04_18_180000_create_webhook_receipts_table` migration.
- Mass `->delete()` is intentional — `WebhookReceipt` does NOT use spatie `LogsActivity`. BaseCommand's correlation_id LogBatch wrapper provides the audit thread for the run itself.

## H-2 — mysqldump command before/after

Sentinel password used in test fixtures (masked here): `SECRET_SHOULD_NOT_LEAK_***`.

**Before (H-2 vulnerability):**
```
mysqldump --single-transaction --skip-lock-tables -h 'wooprod.internal' -u 'msops' -p'SECRET_SHOULD_NOT_LEAK_***' 'meetingstore_woo' | gzip > '/path/to/dump.sql.gz'
```
Visible via `cat /proc/$(pgrep mysqldump)/cmdline` for the entire 1h dump window.

**After (260607-9c6 remediation):**
```
mysqldump --defaults-extra-file='/tmp/msdmp_XXXXXX' --single-transaction --skip-lock-tables 'meetingstore_woo' | gzip > '/path/to/dump.sql.gz'
```
Tempfile contents (chmod-0600, unlinked in `finally{}`):
```
[client]
user=msops
password=SECRET_SHOULD_NOT_LEAK_***
host=wooprod.internal
```

The password is gone from argv. `chmod` runs BEFORE `file_put_contents` so the
tempfile is mode-0600 from the instant it exists on disk. `finally{}` unlinks
on both success and failure paths (T-9c6-04 from the plan threat model).

Sentinel-password technique in the test (`SECRET_SHOULD_NOT_LEAK_123!@#`)
ensures any future regression that reintroduces `-p<pwd>` fails CI loudly via
`expect($snap->capture['cmd'])->not->toContain(SECRET_SENTINEL)`.

## Deviations from plan

**1. Test-file parse error (resolved during Task 3 TDD RED→GREEN).** The
initial copy of the Pest header comment included `/proc/*/cmdline` — the
`*/` substring prematurely closed the `/* ... */` block comment and the
parser exploded with "unexpected token 'for'". Rephrased to `/proc/PID/cmdline`
in both the test docblock and the SnapshotWooDbCommandTest docblock pattern.
No impact on assertions; trivial parser-hygiene fix during RED iteration.

**2. Helper-function return shape (resolved during Task 3 TDD).** The plan
proposed `function makeRecordingSnapshotter(): array` returning
`[$instance, &$instance->capture]` so tests could destructure to
`[$snap, $capture]`. PHP's array-by-reference semantics through a function
return + list-destructure didn't carry the reference cleanly — `$capture`
saw the initial null defaults, not the mid-run writes. Fix: return only the
instance and read `$snap->capture[...]` directly in tests. Functionally
identical; cleaner code.

**3. Explicit command registration (in scope, but worth noting).** Like the
existing `PruneOrphanSuggestionsCommand` and other Domain-namespaced commands,
`PruneWebhookReceiptsCommand` is registered explicitly in
`app/Providers/AppServiceProvider.php` rather than auto-discovered.
Laravel's command auto-discovery only walks `app/Console/Commands/*`, not
`app/Domain/*/Console/Commands/*`. The plan didn't call this out explicitly
but the working precedent was unambiguous.

No other deviations. No architectural changes. No new composer packages.
No env() introduced (EnvUsageTest still green).

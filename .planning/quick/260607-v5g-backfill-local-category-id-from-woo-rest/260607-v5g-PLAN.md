<objective>
Backfill local `products.category_id` + `products.category_ids` for the 3,244 live MS rows where both columns are NULL, by pulling the authoritative `categories` array from Woo REST `GET /wp-json/wc/v3/products?include=...`.

Purpose:
- The legacy `woo:import-products` command (quick 260504-d7v) ingested SKU/name/price/stock but skipped the `categories` array — so the local mirror is incomplete.

- Re-importing via the heavy `woo:import-products` would re-pull EVERYTHING (price, stock, name). This command does the minimum: 2 columns, Woo-as-truth, free, deterministic, no Claude.

Architectural notes (capture inline as comments):
- `DB::table('products')->update(['category_ids' => json_encode($idList)])` is INTENTIONAL — bypassing Eloquent for speed (3,244 rows ≈ 65 chunks of 50). The `array` cast on the model would auto-encode if we used `Product::where(...)->update(...)`, but `DB::table` does NOT — explicit `json_encode` is required. See Product model docblock.
- `orderby=include` is Woo's documented idiom for "preserve the request-order in the response," which lets us avoid an O(N²) reverse-lookup loop. We still build a lookup map (`$lookup[$wooId]`) because Woo may omit IDs that no longer exist on its side (returns silently as `woo_not_found`).
- Cursor-based candidate stream avoids loading 3,244 Product rows at once (Eloquent hydration would be ~10MB+ vs ~1MB for the sku+woo_id map).
- No env() calls. WooClient already handles credentials via IntegrationCredentialResolver.

<objective>
Add per-supplier stale-feed handling so suppliers that go silent for weeks (Nuvias is the named example — they sometimes stop uploading for 3+ weeks while their old prices and stock numbers keep living in `supplier_offer_snapshots`) stop contaminating downstream decisions.

Policy is **stock-decay (option B)**: when a supplier's most-recent snapshot is older than its grace, scanners treat that supplier's stock as 0 but keep its price visible. Mirrors how the legacy Stock Updater plugin behaved.

The fresh/stale decision is centralised in ONE service (`SupplierFreshnessResolver`) per the 260607-pys drift-prevention principle. Three consumers opt in via a constructor flag defaulting TRUE; flipping the policy off (or moving the threshold) is then a one-file edit, never three.

**Three states** per supplier:
- `fresh`   — `MAX(recorded_at) >= NOW() - threshold_days` (default 7)
- `amber`   — within the last `amber_warning_ratio` of the window (default 0.7 → 5-7 days for a 7-day grace) — early-warning, no exclusion yet
- `stale`   — past the threshold; exclusion fires

**Three wiring sites:**
1. `AdCandidateScanner` — in-stock predicate excludes stale-supplier snapshots so the page no longer recommends Google Ads spend on parts only "in stock" at a silent supplier.
2. `CompetitorPositionScanner` — the cheapest-supplier-name resolver (the "Our cost (ex)" sub-line on the bucket popup) skips stale suppliers; falls through to next-cheapest fresh.
3. `SupplierDbSyncCommand::buildBestOfferMap` — the buy-price selector picks `min(price) among offers with stock > 0` from **fresh** suppliers only. (NOTE: brief named `BackfillMerchantFeedCommand` here but that command does not select buy_price — it backfills EAN/brand/category. The actual buy-price resolver is in SupplierDbSyncCommand per 2026-05-25 fix. Plan corrects this.)

**Stale-supplier suppression at the source** (SupplierDbSyncCommand) is the cleanest cut: once buy_price stops absorbing stale-supplier prices, downstream Pricing/Ads/Merchant Feed naturally heal because they all read `products.buy_price`. The two scanner wirings remain because both also read `supplier_offer_snapshots` directly (Ads for stock, Pricing popup for cheapest-supplier name).

**Schedule:** `suppliers:check-stale` Mon-Fri 07:30 London — 30 min after the 07:00 `supplier:db-sync` so the freshness count reflects "today's sync had a chance and didn't show up."

Purpose: stop stale-supplier data silently driving ad spend, pricing decisions, and Merchant Center buy-cost.
Output: one centralised resolver, one snapshot command + cron, three opt-in wiring flags (all defaulting TRUE), one dashboard widget, one notification-centre bucket, and a Pest suite that pins both the resolver math and each scanner's flag-flip behaviour.
</objective>

(1) New `app/Domain/Sync/Services/SupplierFreshnessResolver.php`:
- namespace `App\Domain\Sync\Services`
- final class
- Constructor takes no args (reads `config('supplier.*')` directly)
- Private property `?array $cache = null` for per-request memoisation. First call builds the full classification map; subsequent calls reuse it. (Prevents 3 scanners each querying supplier_offer_snapshots.)
- Driver-aware days-since SQL: MySQL `DATEDIFF(CURDATE(), MAX(recorded_at))`, SQLite `CAST(julianday('now','start of day') - julianday(MAX(recorded_at)) AS INTEGER)`. Resolve driver via `DB::connection()->getDriverName()`.

Public methods:
- `freshSupplierIds(): \Illuminate\Support\Collection<string>` — supplier_ids (strings) with status='fresh'
- `staleSupplierIds(): \Illuminate\Support\Collection<string>` — status='stale'
- `amberSupplierIds(): \Illuminate\Support\Collection<string>` — status='amber'
- `classify(string $supplierId): string` — returns 'fresh'|'amber'|'stale'|'unknown' (unknown = supplier has zero snapshots ever)
- `thresholdDaysFor(string $supplierId): int` — looks up `suppliers.stale_after_days` for this supplier_id; falls back to `config('supplier.default_stale_after_days', 7)`. Cached in the same per-request map.
- `latestRecordedAtFor(string $supplierId): ?\Illuminate\Support\Carbon` — for snapshot writes + UI
- `daysSinceFor(string $supplierId): ?int`

Build the map ONCE in private `loadCache()`:
- Pull every distinct `supplier_id` from `supplier_offer_snapshots` along with `MAX(recorded_at)`. Group BY supplier_id. One query.
- Pull all rows from `suppliers` table for the per-supplier overrides + names. One query.
- For each supplier_id observed: compute days_since, look up override → threshold_days, derive amber boundary as `floor(threshold_days * (1 - config('supplier.amber_warning_ratio', 0.7)))` (e.g. 7d × 30% remaining = within last 3d → amber). Classify.
- Suppliers in `suppliers` table but with zero snapshots → status='unknown'.
- Suppliers in snapshots but not in `suppliers` table → status='unknown' for `thresholdDaysFor` purposes UNTIL upserted by Task 3; classify them using the default threshold so freshness still works on day 1.

Amber math reference (default config):
- threshold_days = 7, amber_warning_ratio = 0.7
- "within last 30% of window" → days_since >= floor(7 * 0.7) = 4
- So days_since ∈ {4, 5, 6} → amber; days_since ∈ {0..3} → fresh; days_since >= 7 → stale.

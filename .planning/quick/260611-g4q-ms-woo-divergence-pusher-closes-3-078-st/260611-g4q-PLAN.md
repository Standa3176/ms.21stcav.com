# 260611-g4q — MS→Woo Divergence Pusher

## Why this exists

- 260610-qc4 grew `WooFieldComparator` from 7 → 13 fields. First live run on prod (2026-06-11) emitted **4,235 sync_diffs across 3,348 products** — true cutover parity is **45%**, not the 100% green the original 7-field scan claimed.
- Field distribution: stock_quantity=**1,598** / buy_price=**1,480** / category_id=**555** / image_url=382 / exists=219 / meta_description=1.
- The first 3 fields are **3,078 (73%)** of the gap AND are pushable MS→Woo with deterministic logic. This task ships the pusher.
- image_url + exists need separate triage (image cache invalidation logic / deleted-Woo-row decision) — out of scope.
- Existing commands won't close this gap:
  - `OverridePopulator` writes `pin_*` flags only, never to Woo, and explicitly skips diffs where `pin_column=null` (which IS our entire 3-field set).
  - `ResyncProductsToWooCommand` pushes regular_price + brand tag + attributes — not stock, not buy_price meta, not category.

## Drift-prevention contract

- The 3 supported fields (`stock_quantity`, `buy_price`, `category_id`) MUST stay aligned with `WooFieldComparator`'s comparable field list.
- This command imports `App\Domain\Cutover\Services\WooFieldComparator` and uses `WooFieldComparator::BUY_PRICE_META_KEY` — **never duplicate** the literal `'_alg_wc_cog_cost'` string.
- A private const `SUPPORTED_FIELDS = ['stock_quantity', 'buy_price', 'category_id']` makes the supported set grep-discoverable; passing `--field=unknown` MUST bail with a clear "unsupported field" error.
- If a future quick task adds a 14th comparable field, the dev MUST either (a) extend SUPPORTED_FIELDS + the payload builder here, OR (b) explicitly out-scope the new field with a comment. `DivergenceComparatorCoverageTest` (260610-qc4) will fail when the comparator changes; this command will refuse unknown `--field=` values until extended.

## Behavioural contract (5 outcomes per product)

| Outcome | Meaning | sync_diff status after run |
|---|---|---|
| `pushed` | All eligible fields landed on Woo via single PUT | applied + applied_at=now |
| `errors` | GET or PUT failed → entire product rolled back | unchanged (pending) — retries on next run |
| `no_woo_product_id` | Product row has woo_product_id=NULL (shouldn't happen post-cutover) | applied + applied_at=now + meta annotation (no retry) |
| `woo_not_found` | Pre-GET returned 404 (product deleted on Woo) | status=woo_not_found (NOT applied — data wasn't pushed; row is retired so it stops surfacing) |
| `already_applied` | sync_diff query already had status='applied' | informational counter only — not surfaced unless query is widened |
| `partial_success` | Reserved for forward compat — single-PUT design means this counter stays at 0 | n/a (counter exists for grep-discoverability) |

## Constraints baked into the design

- **Pre-GET is non-negotiable for any product with a buy_price diff.** Algoritmika WC COG writes `_alg_wc_cog_cost` into Woo's `meta_data[]` array alongside Yoast SEO entries, EAN keys, brand_id meta, etc. A blind PUT with `meta_data=[{key:_alg_wc_cog_cost, value:X}]` WIPES every other meta entry. Must be read-modify-write: GET → splice the existing meta_data → PUT the merged result.
- **Stock-only or category-only diffs:** pre-GET is still cheap insurance (catches woo_not_found 404 cleanly) but technically optional. Design choice: pre-GET always (uniform code path, 1 extra round-trip vs the cost of meta wipe — acceptable).
- **Multi-field combo:** SINGLE PUT carries all 3 fields. NOT 3 separate PUTs (would triple Woo round-trips for 1,000+ products → ~33 min vs ~11 min wall time).
- **WAF split-PUT workaround NOT needed.** 260530-clv's ResyncProductsToWooCommand splits because `regular_price` + everything-else triggers the Cost-of-Goods recompute hook. Our payload (stock + meta + categories) has NO regular_price → no split needed. `WooClient::put()` still routes POST-as-PUT via `services.woo.use_post_for_updates` — leave that to WooClient.
- **Stock quantity needs `manage_stock=true`.** Setting `stock_quantity` without `manage_stock=true` does NOT cause Woo to honor the quantity on the storefront (Woo treats it as a manual override). Always co-emit.
- **Category payload:** prefer `Product.category_ids` JSON array (multi-cat support). Fall back to single `category_id` when `category_ids` is empty/null.
- **Sync_diff lifecycle:** divergence-scan writes status='pending'. This command flips individual rows to status='applied' on successful PUT, status='woo_not_found' on Woo 404, leaves status='pending' on transient errors (so re-runs only re-touch still-pending rows). Idempotent re-runs are cheap.
- **Live-confirmation gate:** if NOT --dry-run AND NOT --no-confirm, prompt "About to PUT N products to live Woo. Continue?" — bail on no. `--no-confirm` exists for cron-style invocation (operator opt-in).
- **200ms usleep between products** matches 260607-v5g + 260609-nku + 260611-f1y pacing.
- `env()` guardrail: any future config gating goes through `config()` in cached-config mode (2026-05-31 lesson). N/A for this task — no .env switches.
- DivergenceScanner, OverridePopulator, ResyncProductsToWooCommand, PushVisibilityToWooCommand left UNTOUCHED.

### Deviation discovered + fixed (Rule 1)

The first test run flagged TWO behaviour issues that required a tiny fix to the
command and a sharpened test assertion:

1. **Case D last_synced_at assertion** — when the product is ALREADY in the target
   OOS state and the supplier offer is stale (ignored), all three changed-checks
   return false, so the command correctly marks the product unchanged and does NOT
   write `last_synced_at`. Original assertion `expect(...)->not->toBeNull()` was
   incorrect (was-already-unchanged semantics). Updated to `toBeNull()` with a
   comment explaining the contract.

2. **Case E --skus case-sensitivity (Rule 1 bug)** — original implementation
   lowercased the `--skus` list before `whereIn('sku', $skuList)`. But
   `products.sku` stores the canonical Woo case (e.g. `HA310-2EP`). Lowercasing
   the input meant `--skus=HA310-2EP` would silently NOT match the product. Fix:
   drop strtolower() from the --skus parse — keep trim() only. The
   supplier_offer_snapshots lookup INSIDE the per-product loop still applies
   strtolower() before keying into the matchKey-form sku column. Two independent
   case rules — both correct now. Fix folded into Task 4 commit (single atomic
   commit per plan rule).

# Quick task 260611-rl4 — `cutover:auto-sync` chain command

## Context

**The operator chore today:** every night, an operator manually runs:

```
php artisan products:hydrate-stock-from-offers   # 260611-qcq — separate 07:20 Mon-Fri cron landing tomorrow
php artisan cutover:divergence-scan --live
php artisan products:push-divergence-to-woo --field=stock_quantity --no-confirm
php artisan products:push-divergence-to-woo --field=buy_price       --no-confirm
php artisan products:push-divergence-to-woo --field=category_id     --no-confirm
```

**This task automates steps 2-5 only** into `cutover:auto-sync`, scheduled nightly 23:00 London. Hydrate stays separate (it has its own 07:20 Mon-Fri cron).

## The chain

```
Phase 1 SCAN     → cutover:divergence-scan --live              (writes sync_diffs + sync_diffs_parity snapshot)
Phase 2 PUSH     → products:push-divergence-to-woo --field=stock_quantity,buy_price,category_id --no-confirm --limit=$max
Phase 3 RE-SCAN  → cutover:divergence-scan --live              (fresh parity_percent — different correlation_id)
Phase 4 REPORT   → counters table + audit log; parity-regression detector
```

## The critical guardrail — parity-regression detection

After phase 3, compare `parity_percent_after` vs `parity_percent_before`:

- If parity **decreased** (after < before), something broke:
  - Comparator predicate is wrong (silently flagging things as diverged that shouldn't be)
  - Woo cache hasn't invalidated (PUTs succeeded but reads return stale data)
  - A different writer to Woo (legacy plugin re-emerged, manual edit during the run)
- Action: audit log `cutover.auto_sync_parity_regression` with both values, **exit code 1**.
- Action: **do NOT roll back the pushed changes** — this is a detection alarm, NOT a rollback mechanism. Operator investigates the next morning.

## Why 23:00 London

- After business close (no operator edits in flight that the comparator could race)
- Before 07:00 supplier:db-sync writes tomorrow's snapshot data
- Low-traffic window for the Woo REST API
- Plenty of headroom for a 5-10min run to complete before midnight
- 22:00 (Fri-only `products:audit-categories`) and 00:30 (`quotes:expire`) are the closest neighbours — no 23:xx slot is taken

## `withoutOverlapping(120)`

Defensive: if a run drags past 2hrs (unusual — capped by `--max-products`), the next firing skips. Prevents two concurrent push runs racing on the same sync_diffs.

4. **Phase 2 PUSH**:

   - If exit != 0 → audit log `cutover.auto_sync_failed` with `['phase' => 'rescan', 'exit_code' => $exitCode, 'parity_before' => $parityBefore]`, return 1.

### Deviation — Rule 1 (Bug fix folded in)

**Found during:** Case A (Logitech happy-path) initial red.

**Issue:** Original `SKU_LIKE_PATTERN = '/^[A-Z0-9][A-Z0-9\-#]+$/i'` matched pure-alphabetic 6+ char tokens like "Logitech" / "Microsoft" / "Vivitek" as SKU-shaped, so the first word was skipped and the second word ("4k", "conference", etc.) was tried — falsely classifying every long-named brand as a SKU-like prefix.

**Brief language:** "uppercase + digits + hyphen/#, length > 5". The "digits + hyphen/#" half was load-bearing and the implementation had drifted to "alphanumeric + hyphen/#".

**Fix:** `SKU_LIKE_PATTERN = '/^(?=.*[0-9\-#])[A-Z0-9][A-Z0-9\-#]+$/i'` — positive lookahead ensures the token MUST contain at least one digit, hyphen, or '#'. Single-pass regex. "AV1E3AA#AC3" still matches (contains digits + '#'); "Logitech" no longer matches (pure letters).

**Files modified:** `app/Console/Commands/BackfillProductBrandFromNameCommand.php` (SKU_LIKE_PATTERN const + docblock note).

**Commit:** 5c19c7a (folded into the test commit so the test that catches the bug ships alongside the fix).

# Quick 260613-o33 — Fix BrandDuplicateFinder Canonical Selection

## One-liner

Replaced `BrandDuplicateFinder`'s count-based canonical-brand ranking with slug-quality ranking (exact base-slug → clean → numeric-suffix → `-brand`-suffix, tie-break lowest id) so re-runs of `brands:dedupe` against today's prod dataset pick the clean term as canonical, not the `-brand`-suffix duplicate.

## What changed

### `app/Domain/Sync/Services/BrandDuplicateFinder.php`

**Before:** `usort` on `count DESC, id ASC`. `count` was Woo's stale taxonomy cache field. At probe time the `-brand`-suffix dups had higher counts than the clean originals → service told the operator the wrong row was canonical.

**After:**

- `use Illuminate\Support\Str` added for `Str::slug()`.
- The pagination loop now also reads `slug` off each Woo brand row (`'slug' => (string) ($row['slug'] ?? '')`). `count` is still read and propagated through the return rows for display/audit purposes (DedupeBrandsCommand's dry-run table still consumes it).
- The canonical-pick `usort` now ranks by `rankSlug($row['slug'], Str::slug($groupKey)) ASC`, tie-break `id ASC`.
- New private `rankSlug(string $slug, string $expectedBaseSlug): int` method with the ranking semantics below.
- Inline `2026-06-13 INCIDENT` comment above the canonical-select block; class-level docblock incident paragraph; `discover()` PHPDoc `@return` updated to reference `rankSlug()` and add `slug:string` to row shapes.

**Slug rank semantics:**

| Rank | Meaning              | Example (for name 'Barco', baseSlug='barco') |
| ---- | -------------------- | -------------------------------------------- |
| 0    | exact base-slug      | `barco`                                      |
| 1    | clean non-base       | `barcoshop` (no numeric, no -brand)          |
| 2    | WC dedup numeric     | `barco-1`, `barco-2`                         |
| 3    | legacy `-brand`      | `barco-brand`                                |

Lower = better.

**Public API contract preserved:** Consumers (`DedupeBrandsCommand`, `RetagProductsOnWooCommand`) read only `$entry['canonical']['id']` and `$entry['sources'][]['id']` — both ignore unknown row fields, so adding `slug` to the row arrays is fully backwards-compatible. Both regression suites confirm this (see Metrics).

### `tests/Unit/Domain/Sync/Services/BrandDuplicateFinderTest.php` (NEW)

7 Pest cases A-G with an anonymous-subclass WooClient stub mirroring the pattern from `DedupeBrandsCommandTest`:

- **A** — Clean slug beats `-brand` suffix (Yealink 3001 vs 12776).
- **B** — Numeric suffix beats `-brand` suffix (Yealink 12777 vs 12776).
- **C** — Case-mismatch still groups under `strtolower`; exact base-slug wins.
- **D** — `rankSlug` pure-function asserts via Reflection (barco=0, barco-brand=3, barco-1=2, barcoshop=1).
- **E** — Singletons dropped (Apple/Sony/Logitech each appears once → empty plan).
- **F** — Pagination across 3 pages still groups Poly/Bose/Logitech correctly.
- **G (REGRESSION)** — today's exact 11-brand prod dataset (Barco/Crestron/LG/Neat/Yealink, each with a `-brand`-suffix dup carrying `count=99` and the clean slug carrying `count=1`). Asserts canonical IDs are the clean-slug ones: **3102, 3012, 2904, 3096, 3001** — NOT 13033, 12781, 12779, 12777, 12776.

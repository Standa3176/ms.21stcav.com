# Phase 260626-d76 Plan 01: Fix prod 500 on /admin/category-audit (MariaDB 1235) Summary

Fixed the production 500 on `/admin/category-audit` by replacing a `whereIn(subquery + LIMIT)` latest-run scope with a MariaDB-safe scalar-resolution + equality predicate, and added a driver-portable SQL-shape regression guard so the SQLite test suite can now catch this MariaDB-only incompatibility.

## What changed

### `CategoryAuditPage::buildQuery()`

**Before** (app/Filament/Pages/CategoryAuditPage.php:215-225):

```php
private function buildQuery(): Builder
{
    return CategoryAuditFinding::query()
        ->with('product:id,name')
        ->whereIn('run_id', function ($q): void {
            $q->from('category_audit_findings')
                ->select('run_id')
                ->orderByDesc('audited_at')
                ->limit(1);
        });
}
```

**After:**

```php
private function buildQuery(): Builder
{
    $latestRunId = CategoryAuditFinding::query()
        ->orderByDesc('audited_at')
        ->value('run_id');

    return CategoryAuditFinding::query()
        ->with('product:id,name')
        ->when(
            $latestRunId !== null,
            fn (Builder $q): Builder => $q->where('run_id', $latestRunId),
            fn (Builder $q): Builder => $q->whereRaw('1 = 0'),
        );
}
```

The docblock above `buildQuery()` was also updated to document the scalar-resolution approach and the MariaDB IN+LIMIT (error 1235) incompatibility, replacing the old "subquery on run_id" wording.

## Why the bug was invisible to the test suite

The test DB is SQLite (`:memory:`), which **permits** `IN (subquery + LIMIT)`. Production runs MariaDB, which **rejects** it:

> `SQLSTATE[42000]: 1235 This version of MariaDB doesn't yet support 'LIMIT & IN/ALL/ANY/SOME subquery'`

Filament's pagination count query (`getCountForPagination`) compiles the table query — including the latest-run scope — on every render, so the malformed `IN(subquery + LIMIT)` tripped error 1235 and returned a 500. Pest stayed green throughout because SQLite never raised the error.

### How the new SQL-shape guard closes the gap

The new regression case `it('buildQuery does not use an IN-with-LIMIT subquery (MariaDB 1235 guard)')` inspects the compiled SQL via `->toSql()` and asserts it contains **no** `in (select … limit)` construct (`preg_match('/in \(select.*limit/is', $sql)` must be `0`), plus a positive assertion that the scope is an equality predicate on `run_id`. This is a driver-portable assertion — it runs on SQLite but catches the MariaDB-only incompatibility. It RED'd against the old `whereIn(subquery LIMIT)` implementation and is GREEN against the fix.

Two further cases lock the behaviour:
- `it('shows only the latest run\'s findings when multiple runs exist')` — seeds the latest run plus an inline older-run record (run_id `01HW…`, `audited_at => now()->subDay()`) and asserts only the latest-run SKU is visible.
- `it('buildQuery returns zero rows when the table is empty (no error)')` — proves the `whereRaw('1 = 0')` empty-table guard.

<objective>
Stop the auto-reprice failing (and cluttering failed_jobs) on products whose Woo product was deleted.
Blast-radius scan (2026-07-01): of 6,237 products with a woo_product_id, 204 are STALE (Woo returns no
such id) and ALL 204 are status=draft (old `manual` imports). Live shop is fine — every PUBLISHED
product has a valid id, so live prices sync correctly. The failures are noise: PushPriceChangeToWoo
PUTs a price for these drafts, Woo replies `woocommerce_rest_product_invalid_id`, the job throws +
retries.

FIX (low-risk, error-path + cleanup only):
  1. PushPriceChangeToWoo skips products where status != 'publish' (drafts aren't on the shop).
  2. On a Woo invalid-id error, catch → log → NULL the stale woo_product_id → return (no rethrow), so
     the job stops failing and the product is flagged for re-link. Other errors still rethrow (retry).
  3. products:reconcile-stale-woo-ids to clean the existing 204 in one pass (--dry-run first).
Working live-price pushes are untouched.
</objective>

CHANGES:
- After the existing null-woo-id skip, add a status guard:
  ```php
  if ($product->status !== 'publish') {
      Log::info('pricing.woo_push_skipped_not_published', [
          'product_id' => $product->id, 'sku' => $event->sku, 'status' => $product->status,
      ]);
      return;
  }
  ```
- Wrap BOTH `$this->woo->put(...)` calls through a private helper that catches the WC invalid-id:
  ```php
  private function putOrClearStale(string $path, string $regularPrice, Product $product, ProductPriceChanged $event): void
  {
      try {
          $this->woo->put($path, ['regular_price' => $regularPrice]);
      } catch (\Throwable $e) {
          if (str_contains($e->getMessage(), 'woocommerce_rest_product_invalid_id')) {
              Log::warning('pricing.woo_push_stale_id_cleared', [
                  'product_id' => $product->id, 'sku' => $event->sku,
                  'woo_product_id' => $product->woo_product_id, 'path' => $path,
              ]);
              $product->forceFill(['woo_product_id' => null])->saveQuietly();
              return; // stale link cleared + flagged; do NOT rethrow (no retry, no failed_jobs)
          }
          throw $e; // genuine/transient — let the job retry
      }
  }
  ```
  Call it for the variant path (`products/{pid}/variations/{vid}`) and the product path
  (`products/{pid}`). (Nulling woo_product_id is the right recovery even on the variant path — an
  invalid product id makes the variation moot.)
- `str_contains` on the WC error CODE is robust: the observed exception message is
  `... Error: Invalid ID. [woocommerce_rest_product_invalid_id] ...`.

Product model: has `status` ('publish'|'draft'|'pending'), `woo_product_id` (nullable), `sku`.

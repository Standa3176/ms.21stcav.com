<objective>
Foundation for the operator-confirmed "Brands to Add" workflow (Piece 1 of 2; Piece 2 = the page +
Suggestions filters + one-click Create-on-Woo). The brand of a new_product_opportunity is NOT stored on
the suggestion — it comes from the supplier feed manufacturer — so filtering/grouping by brand needs a
precompute. This task builds that precompute: resolve each pending SKU's brand, tag the suggestion
(evidence.brand + evidence.brand_on_woo) so Piece 2's list filters are cheap, and cache a brands-to-add
summary (brand => products-it-would-unlock) so Piece 2's page renders instantly. Also extracts the brand
resolution into a shared trait so this command and DraftFromSuggestionsCommand share one implementation.
No storefront writes, no Claude.
</objective>

Rules per sku (lowercased key):
 - manufacturers empty / not in map => sourceable=false, brand=null, on_woo=false (NOT in to_add).
 - firstResolvableBrandKey(mfrs, woo) returns [bk,mfr] non-null => on_woo=true, brand=canonical woo name
   ($wooBrandsByLower[$bk]), sourceable=true.
 - has manufacturer(s) but none resolve => on_woo=false, sourceable=true, brand = the FIRST manufacturer
   (the thing to add); add to to_add[brand]: count++ and push sku (cap sample skus at ~25).

RefreshBrandsToAddCommand.perform() (signature: `products:refresh-brands-to-add {--dry-run} {--limit=0}`):
 1. Load $wooBrandsByLower (abort if empty).
 2. Walk pending suggestions → collect [suggestionId => sku]. Batch supplier_products → $skuToManufacturers.
 3. $index = $this->buildBrandsToAddIndex($skuToManufacturers, $wooBrandsByLower).
 4. If NOT dry-run: for each suggestion, merge into evidence: brand = $index['per_sku'][lower(sku)]['brand'],
    brand_on_woo = ...['on_woo'] (default false / null when sku absent); $suggestion->update(['evidence'=>$ev]).
    Cache::put('suggestions.brands_to_add', ['generated_at'=>now()->toIso8601String(), 'brands'=>ksorted
    to_add as list of {brand,count,skus}], 86400).
 5. Print the to-add summary table (brand | count) always.

<objective>
The 260702-qd8 auto-create-brand feature guards against creating a brand that already exists — but only by
EXACT (case-insensitive) name match. Feed manufacturers are frequently more specific than the clean Woo
brand ('Barco Clickshare' vs 'Barco', 'HP Inc' vs 'HP', 'Sony Professional' vs 'Sony'), so on the
bulk/auto-create path WooBrandCreator would POST a near-duplicate brand next to the clean one — the exact
taxonomy pollution the brand cleanup fixed. The per-row approve path already avoids this because it
fuzzy-matches via TaxonomyResolver::resolveBrand first. Fix: make WooBrandCreator's existence check reuse
that SAME fuzzy matcher, so a more-specific name reuses the existing brand and only genuinely-new names get
a new term.
</objective>

<interfaces>
WooBrandCreator already injects TaxonomyResolver $taxonomy. Replace the exact-loop findExistingId with a
delegation to the fuzzy resolver (bestMatchId, threshold 0.85 — containment like 'barco'⊂'barco clickshare'
scores 0.9 so it matches; genuinely different names stay below 0.85 and don't):

```php
/**
 * Find an EXISTING Woo brand id for this (normalised) name, or null. Uses the
 * same fuzzy matcher the per-row create path trusts (TaxonomyResolver::resolveBrand
 * → bestMatchId, FUZZY_THRESHOLD 0.85) so a more-specific feed name like
 * 'Barco Clickshare' reuses the existing 'Barco' term instead of spawning a
 * near-duplicate. resolveBrand normalises + reads the cached allBrands() list.
 */
private function findExistingId(string $name): ?int
{
    $id = $this->taxonomy->resolveBrand($name);
    return ($id !== null && $id > 0) ? $id : null;
}
```
Everything else in ensureBrandTermId stays: normalise → junk/blank → null; findExistingId hit → return it
(no POST); else POST + shadow-guard + cache-forget + return new id; term_exists → forget + findExistingId.
No signature changes. No other files.

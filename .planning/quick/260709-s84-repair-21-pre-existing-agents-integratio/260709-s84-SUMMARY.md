## Buckets — root cause + fix

### Bucket 1 — Driver-portability (7 failures) — **CODE (the one real fix)**
`AgentRunGdprScrubber::scrubForCustomer()` and `AgentsGdprPurgeLangfuseCommand::perform()` used
`whereRaw('JSON_SEARCH(LOWER(tool_calls), ?, ?) IS NOT NULL', ['one', $email])`. `JSON_SEARCH` is
MariaDB-only and absent on SQLite, so the whole query threw on the test DB (the scrubber was therefore
effectively untested). Replaced with the driver-portable `whereRaw('LOWER(tool_calls) LIKE ?', ['%'.$email.'%'])`,
keeping the sibling `orWhere('agent_reasoning_summary', 'like', ...)` and the OR grouping exactly.

- **Type:** production code fix (the only prod behaviour change in this plan)
- **Behaviour delta:** LIKE is a **broader-or-equal** match than JSON_SEARCH — it can only match *more* rows, never fewer. For GDPR erasure that is safer (never misses an email-bearing row). Runs identically on SQLite (CI) and MariaDB (prod). GDPR erasure is now also testable on SQLite CI.
- **Ref:** [[meetingstore-sqlite-mariadb-strict-trap]] — the SQLite↔MariaDB portability class of bug bit again.

### Bucket 2 — Missing test credential (7 failures) — **TEST-ROT**
`ClaudeClient` resolves the Anthropic api_key via `IntegrationCredentialResolver`, which throws
`IntegrationCredentialMissingException('anthropic_api')` when neither a DB row nor the env fallback
`config('prism.providers.anthropic.api_key')` is set. `RunPricingAgentJobTest`, `ClaudeClientTest`,
and `PricingAgentCalibrationTest` provisioned neither. Fixed by setting the env fallback in each setup
(`config()->set('prism.providers.anthropic.api_key', 'test-key')`), mirroring the passing
`ClaudeClientResolverIntegrationTest`. `Prism::fake()` still intercepts the real HTTP, so the value is inert.
`ClaudeClientTest` also needed `Context::add('correlation_id', ...)` because the `integration_events`
row it writes has a NOT NULL `correlation_id`.

### Bucket 3 — In-test queued-listener serialization (3 failures) — **TEST-ROT**
`RunPricingAgentJobTest` fixtures 2, 3 & 4 call the job's `handle()` directly. `handle()` fires
`AgentRunStarted`/`AgentRunCompleted`, whose real `ShouldQueue` `NotifyOn*` listeners tried to serialize
in-test → *"Serialization of Closure is not allowed."* Added `Queue::fake()` to the `beforeEach` so the
queued listeners are recorded, not serialized. This is a test-env artifact (prod listener deps are
container-resolved/serializable), not a prod bug — verified: `Queue::fake()` fully resolved it, so it was
**not** a real event-carries-closure issue. Fixture 1's existing `Event::fake(...)` + `assertDispatched`
assertions are preserved and unaffected.

### Buckets 4 + 5 — Enum outgrew its tests (3 + 1 failures) — **TEST-ROT**
`IntegrationCredentialKind::cases()` now returns **10** (grew past the 7 the enum had when the tests were
written — Icecat, EanSearch, ImageSearch, OpenAiApi added since). `IntegrationCredentialKindEnumTest`
hard-coded `7`; `IntegrationHealthWidgetTest` hard-coded `5` tiles in three places. Both updated to
**derive** the expected count from `count(IntegrationCredentialKind::cases())` so they track the enum and
never rot again, rather than swapping one magic number for another.

## Deviations from Plan

### Discovery — AgentRunGdprScrubberTest Test 3 timestamp assertion (test-rot, unmasked by Bucket 1)
After the Bucket 1 fix, 7/8 GDPR tests went green but Test 3 failed on
`expect($run->started_at->equalTo($started))->toBeTrue()` (line 75). Root cause: `agent_runs.started_at`/
`completed_at` are `timestamp` columns (second precision, no microseconds), while the factory's in-memory
Carbon carries microseconds — so `equalTo()` fails on the DB roundtrip. This is **driver-agnostic** (fails
on MariaDB too — `timestamp` has fractional precision 0) and was **masked** because the JSON_SEARCH query
threw before line 75 was ever reached. Verified empirically via tinker (`.369434` in-mem vs `.000000`
persisted). It is NOT JSON_SEARCH-specific and NOT a production bug (the scrubber never writes these
columns). Fixed minimally by comparing at the precision the column actually persists
(`->format('Y-m-d H:i:s')`), preserving the test's intent (timestamps preserved through scrub). This is
the same family of test-rot as the plan's other buckets.

### Out of scope (per plan) — SuggestionResource JSON_SEARCH (latent follow-up)
`SuggestionResource` also uses `JSON_SEARCH`. Its tests pass today (driver-guarded or not hit on SQLite),
so it was left untouched per the plan. **Recommended follow-up:** a portability pass to convert it to the
same `LOWER(col) LIKE` pattern before it bites on a future SQLite-run code path.

### Pre-existing pint issues in touched test files (left untouched — out of scope)
`pint --test` flags `fully_qualified_strict_types`/`ordered_imports` (AgentRunGdprScrubberTest,
IntegrationHealthWidgetTest) and `no_blank_lines_after_phpdoc` (IntegrationCredentialKindEnumTest).
Verified against `HEAD` that these are **100% pre-existing** (on import/docblock lines I did not edit),
not regressions from this plan. Per the plan's scope (pint verification was scoped to the 2 app files,
which are clean) and the executor SCOPE BOUNDARY rule, these were left as-is to avoid noisy reformatting
of unrelated lines.
